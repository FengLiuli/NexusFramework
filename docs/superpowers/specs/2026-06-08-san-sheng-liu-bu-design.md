# 三省六部 Workflow 系统设计

> **For agentic workers:** 本系统由 shangchao SKILL.md 触发，通过 workflow 脚本编排 8 个独立 subagent 构成完整的 AI 编码治理流程。

**Goal:** 将 AI 编码工作流组织为三省（决策-审核-执行）六部（专业分工）的治理体系，支持快速/标准/严谨三档运行模式。

**Architecture:** 以 Workflow 脚本作为编排层，8 个独立 subagent（.agent.md）作为执行层。SKILL.md 负责识别"上朝"关键词并启动 workflow。每个部门在每次执行后积累结构化的部门记忆（经验），供后续同类事务参考。

## 部门记忆系统

### 存储结构

每个部门的记忆存放在独立的 Markdown 文件中，以 `---` 分隔的条目构成经验序列。

```
.claude/
└── memories/
    └── san-sheng-liu-bu/
        ├── zhongshu-sheng.md      # 中书省记忆
        ├── menxia-sheng.md         # 门下省记忆
        ├── shangshu-sheng.md       # 尚书省记忆
        ├── libu.md                 # 吏部记忆
        ├── hubu.md                 # 户部记忆
        ├── li3bu.md                # 礼部记忆
        ├── bingbu.md               # 兵部记忆
        ├── xingbu.md               # 刑部记忆
        └── gongbu.md               # 工部记忆
```

### 记忆条目格式

每条记忆是结构化的 YAML+Markdown 条目：

```yaml
---
date: 2026-06-08
task: 实现 GAS Effect 叠加系统
mode: 严谨
judge: 皇帝（用户）
summary: 完成 Effect 堆叠逻辑，涉及 5 个文件
key_points:
  - Effect 堆叠采用配置优先级而非注册顺序
  - 发现一个 ECS Entity 生命周期隐性依赖
  - 本次方案经过了 2 轮封驳
result: approved
refs:
  - docs/superpowers/plans/2026-06-08-effect-stacking.md
---
```

### 各部门记忆的内容差异

| 部门 | 记忆什么 | 复用场景 |
|------|----------|----------|
| 中书省 | 过往方案的架构取舍、哪些方案被退回、什么原因 | 新任务时避免踩同样的坑 |
| 门下省 | 封驳判例、常见问题模式 | 同类审查可提速，知道"上次这个问题也是这里出问题" |
| 尚书省 | 执行经验、子任务拆分模式 | 类似任务可直接套用已验证的拆分模板 |
| 吏部 | 哪个 agent 擅长什么、哪种调度模式效率高 | 下次同类任务派更合适的人 |
| 户部 | 各类任务的 token 消耗基准 | 新任务进来可预估成本 |
| 礼部 | 频繁违规的模式、架构一致性历史 | 新代码审查时重点检查历史问题点 |
| 兵部 | 过去发现的安全漏洞类型 | 新审查中优先排查已知漏洞模式 |
| 刑部 | Bug 根因分类、修复策略有效性 | 新 Bug 出现时快速匹配相似案牍 |
| 工部 | 技术债记录、组件耦合情况 | 新功能规划时告知哪些地方"动不得" |

### 记忆的使用时机

1. **朝会简报阶段** — 快速扫描相关部门记忆，看是否有相关历史经验
2. **部门执行前** — 读取本部门记忆，了解过往模式
3. **部门执行后** — 写入本次经验，形成闭环
4. **封驳/复审时** — 门下省可查阅过往封驳判例，保持一致的标准

### 记忆的生命周期

- 项目级记忆存放在 `.claude/memories/san-sheng-liu-bu/` 下，随项目持久化
- 不需要版本控制，但推荐 git 跟踪以便回溯
- 记忆条目会随时间增长，户部负责在 budget 报告中提醒记忆膨胀

## 朝会流程

用户喊"上朝"后，触发 SKILL.md → 启动 Workflow：

1. **朝会** — 快速扫描相关记忆，呈报任务简报，列出涉及文件、预估规模、已知风险、历史相关案例
2. **皇帝定夺** — 用户选择：快速 / 标准 / 严谨
3. **进入对应流程**

### 三档运行模式

| 模式 | 参与部门 | 适用场景 |
|------|----------|----------|
| 快速 | 仅工部 | Bug 修复、简单修改、已知方案 |
| 标准 | 中书省 → 门下省 → 尚书省(工部) | 功能开发、模块扩增 |
| 严谨 | 三省 + 六部 + 门下省复审 + 对抗验证 | 架构变更、大规模重构、高风险改动 |

## 封驳机制（门下省）

```
门下省收到方案/代码
  → 查阅过往封驳判例
  → 审查是否合格
    → 合格：通过，进入下一环节
    → 不合格：封驳退回，附修改意见
      → 最多 3 轮封驳
      → 第 3 轮仍未通过 → 升级给用户裁决
  → 记录本次审查到部门记忆
```

封驳注意事项：
- 小问题：仅附修改意见，不消耗轮次
- 中等问题：记录封驳轮次 + 详细修改说明
- 大问题：封驳 + 建议重新拟旨

## 六部并行模式

严谨模式下，尚书省收到中书省方案后：

```
1. 礼部 + 兵部 → parallel 并行审查规范和安全性（作为前置检查）
2. 吏部 拆分任务（参考过往调度经验） → 工部 实施
3. 刑部 编写/执行测试（与工部并行）
4. 户部 全程监控 budget
5. 各部执行结束后写入本部门记忆
```

标准模式下仅有工部参与尚书省阶段。

│   ├── hubu.agent.md                # 户部

│   ├── bingbu.agent.md              # 兵部

### 关键技术决策

1. **agent.md 名称使用拼音** — 避免中文路径问题，同时对英文读者可读
2. **礼部使用 li3bu** — 区别于吏部的 libu，避免 agent 名称冲突
3. **SKILL.md 为薄入口** — 只做关键词匹配 + 启动 workflow，所有逻辑在 workflow 脚本和 agent 文件中
4. **封驳循环上限 3 轮** — 防止死循环，3 轮后升级给用户
5. **户部不单独派 agent** — budget 管理通过 workflow 脚本的 `budget` 全局变量内联处理，不需要额外 agent 调用
6. **记忆为结构化 YAML+Markdown** — 每次 agent 执行结果（关键信息 + 自然语言总结）追加到对应记忆文件
7. **记忆读取在朝会阶段** — 不每次调用都读，只在朝会阶段扫描历史，减少 token 浪费

### Workflow 编排伪代码

```
朝会阶段:
  scan memories → 是否有关联历史
  agent() → 呈报简报，输出 { mode, task }

中书省阶段 (中/严):
  read zhongshu-sheng memories (相关条目)
  agent('zhongshu-sheng') → 输出 { plan }
  append to zhongshu-sheng memories

门下省审查 (中/严):
  read menxia-sheng memories (封驳判例)
  agent('menxia-sheng') → 审查方案
  loop 封驳 (最多 3 轮)
  append to menxia-sheng memories

尚书省执行:
  快速: agent('gongbu') → 直接实施
  严谨:
    parallel([agent(礼部), agent(兵部)])
    → 吏部 拆分任务 → parallel([工部实施, 刑部测试])
    → agent(门下省复审)
  append to 各部 memories

复命:
  agent() → 汇总结果 + 本次执行摘要(to memory), 奏报御览
```

## 设计决策记录

| 决策 | 选项 | 选择理由 |
|------|------|----------|
| 触发词 | "上朝" / "开个朝会" | 简短自然，符合历史语境 |
| 规格选择 | 朝会上由用户定 | 保留灵活性，不预设 |
| 每个职位独立 agent | Workflow script vs agent.md | agent.md 封装专业知识和工具权限更干净 |
| 命名 | 拼音 | 兼容文件系统，对中英文都可读 |
| 封驳上限 | 3 轮 | 防止死循环，给用户保留最终裁决权 |
| 户部实现 | 内联 budget | 不需要独立 agent，workflow 已有 budget API |
| 记忆格式 | YAML frontmatter + Markdown | 结构化可解析，同时保留自然语言描述 |
| 记忆读取时机 | 朝会阶段批量扫描 | 避免每次 agent 调用都读记忆，减少 token 浪费 |
